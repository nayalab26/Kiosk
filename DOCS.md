# Киоск — Документация проекта

## Что такое Киоск
Киоск — агрегатор Telegram-каналов в формате Telegram Mini App. Пользователь выбирает интересные темы и получает персональную ленту постов из каналов. Владельцы каналов могут подать заявку на размещение. Монетизация — трафик между каналами и платное размещение.

**Планы на будущее:**
- Версия для мессенджера Max (VK) — после завершения TG версии
- ИИ агент для автоматического создания каналов в Max, парсинга TG и публикации контента

## База данных (Supabase)

### Таблица `applications` — заявки каналов
| Колонка | Тип | Описание |
|---------|-----|---------|
| id | SERIAL | Первичный ключ |
| handle | TEXT | Username канала (без @) |
| title | TEXT | Название канала |
| description | TEXT | Описание канала |

| status | TEXT | pending / approved / rejected |
| chat_id | BIGINT | Telegram chat_id владельца |
| created_at | TIMESTAMP | Дата подачи заявки |

| Колонка | Тип | Описание |
|---------|-----|---------|
| id | SERIAL | Первичный ключ |
| username | TEXT | Telegram username (UNIQUE) |
| chat_id | BIGINT | Telegram chat_id |

| Колонка | Тип | Описание |
|---------|-----|---------|
| id | SERIAL | Первичный ключ |

| post_url | TEXT | Прямая ссылка на пост |
| published_at | TIMESTAMP | Дата публикации |
| created_at | TIMESTAMP | Дата добавления в БД |

## Архитектура Mini App (index.html)

### Экраны
- `screen-onboarding` — выбор тем при первом входе
- `screen-feed` — главная лента постов
- `screen-channel` — страница отдельного канала
- `screen-settings` — настройки тем
- `screen-register` — форма добавления канала
- `screen-profile` — профиль пользователя (уровень, заявки, настройки)

### Ключевые константы в коде
```javascript
const BOT_TOKEN = '...'        // Токен бота (используется для getChat/getChatMemberCount)
const ADMIN_ID = '238292'      // ID администратора
const SUPABASE_URL = '...'     // URL Supabase
const SUPABASE_KEY = '...'     // Publishable key (безопасен для фронтенда)
```

### Логика ленты
1. Загружает одобренные каналы из Supabase (`applications?status=eq.approved`)
2. Фильтрует по выбранным темам пользователя
3. Загружает последние посты из таблицы `posts`
4. Показывает один последний пост на канал
5. Если реальных постов нет — показывает sample посты как fallback

### Хранение данных пользователя
Используется Telegram CloudStorage (CloudStorage.setItem/getItem).
Fallback — localStorage для тестирования вне Telegram.
Хранится: выбранные темы (`kiosk_topics`).

### Темы (категории) — текущие

### Форма регистрации канала
1. Пользователь вводит @username канала
2. Автоматически подтягиваются данные через `getChat` и `getChatMemberCount`
3. Выбирает категории (мультиселект)
4. Вводит контакт для связи
5. Нажимает "Подать заявку" → Mini App вызывает `Telegram.WebApp.sendData(payload)`
6. Бот получает данные, сохраняет в Supabase, запоминает chat_id
7. Админу приходит уведомление с кнопками Одобрить/Отклонить

## Архитектура бота (bot.py)

### Команды
| Команда | Доступ | Описание |
|---------|--------|---------|
| /start | Все | Открыть Mini App |
| /help | Все | Помощь |
| /applications | Только админ | Список pending заявок |
| /approve username | Только админ | Одобрить канал |
| /reject username | Только админ | Отклонить канал |
| /remove username | Только админ | Удалить канал и его посты |
| /sync | Только админ | Ручная синхронизация постов |

### RSS парсер
- Источник: `https://tg.i-c-a.su/rss/{handle}`
- Синхронизация каждые 10 минут (periodic_sync)
- При одобрении канала — сразу запускается первичная синхронизация
- Сохраняет до 5 последних постов на канал
- Проверяет дубликаты по `post_url`

### Логика заявок
1. Владелец заполняет форму в Mini App
2. Фронтенд вызывает `/api/notify` на Railway (прокси)
3. Бот проверяет дубли: если `pending` → "уже на рассмотрении", если `approved` → "уже в Киоске", если `rejected` или нет → создаёт заявку
4. Сохраняет в `applications` (status=pending) с chat_id
5. Админу приходит уведомление с inline кнопками Одобрить/Отклонить
6. Approve → статус approved, запуск RSS синхронизации, уведомление владельцу, удаление дублей
7. Reject → статус rejected, уведомление владельцу, удаление дублей

### Поиск chat_id для уведомлений
1. Поле `chat_id` в таблице `applications`
2. Таблица `users` по `username`
3. Если не найден — сообщает админу контакт для ручной связи

## Процесс деплоя

### Mini App (GitHub Pages)
1. Обновить `index.html`
2. Загрузить на GitHub
3. GitHub Pages автоматически обновит (~2 мин)

### Бот (Railway)
1. Обновить `bot.py` и/или `requirements.txt`
2. Загрузить на GitHub
3. Railway автоматически пересобирает

> ⚠️ Важно: убедиться что запущен только ОДИН деплой в Railway.
> Два активных деплоя вызывают ошибку: `Conflict: terminated by other getUpdates request`

## Правила для разработчиков

1. **Никаких секретов в коде** — все ключи только в Railway Variables
2. **Один файл фронтенда** — весь Mini App в одном `index.html`
3. **Один деплой в Railway** — проверять после каждого пуша
4. **Тестировать в Telegram** — Mini App корректно работает только внутри Telegram на мобильном
5. **Supabase RLS** — все новые таблицы должны иметь политики безопасности
6. **Кириллица в Python строках** — избегать многострочных строк с кириллицей (вызывает SyntaxError)
